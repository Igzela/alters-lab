## 1. Purpose

The Day 30 demo proves that CYCLE-001 can execute a **controlled first cycle** with:

- A working **FastAPI Snapshot Intake API** (in-memory sessions, endpoint-driven)
- **Governed active YAML artifacts** in `alters/current/` produced under human confirmation
- **Explicit YAML export** via service code, not magic writes
- A **complete evidence trail** from API session to YAML file to reality trace

The demo does **not** claim: LLM-powered dialogue, runtime calibration, drift computation, database persistence, frontend UI, or production deployment.

## 3. Demo Flow — 9 Steps

The demo executes a linear 9-step flow. Each step has a clear pass condition before proceeding.

### Step 1: Start API

```bash
cd apps/api
uvicorn alters_lab.main:app --host 0.0.0.0 --port 8000
```

**Pass**: Server starts without errors, listens on port 8000.

### Step 2: Health Check

```bash
curl http://localhost:8000/health
```

**Expected response**:
```json
{"status": "ok", "service": "alters-lab-api"}
```

**Pass**: 200 OK, `status: "ok"`.

### Step 3: Create Session

```bash
curl -X POST http://localhost:8000/snapshot-intake/sessions
```

**Expected**: 201 with `session_id` (UUID), empty anchors, `intake_status.phase: "asking_heaviest_constraint"`.

**Pass**: Session created, `pending_anchor: "heaviest_constraint"`.

### Step 4: Submit 3 Anchors (sequential)

```bash
# Anchor 1: heaviest_constraint
curl -X POST http://localhost:8000/snapshot-intake/sessions/{id}/answers \
  -H "Content-Type: application/json" \
  -d '{"anchor": "heaviest_constraint", "answer": "..."}'

# Anchor 2: most_unclear
curl -X POST http://localhost:8000/snapshot-intake/sessions/{id}/answers \
  -H "Content-Type: application/json" \
  -d '{"anchor": "most_unclear", "answer": "..."}'

# Anchor 3: unwilling_to_give_up
curl -X POST http://localhost:8000/snapshot-intake/sessions/{id}/answers \
  -H "Content-Type: application/json" \
  -d '{"anchor": "unwilling_to_give_up", "answer": "..."}'
```

**Pass**: Each answer accepted (200), `pending_anchor` advances in order, after 3rd answer `pending_anchor: null`.

### Step 5: Confirm Snapshot

```bash
curl -X POST http://localhost:8000/snapshot-intake/sessions/{id}/confirm
```

**Expected**: 200 with `snapshot.intake_status.phase: "completed"`, `ready_for_branch_discovery: true`.

**Pass**: Snapshot confirmed in-memory. **No YAML file written at this step.**

### Step 6: Export to Temp YAML

```bash
# Call write_snapshot_yaml via a temporary script or interactive Python:
python -c "
from pathlib import Path
from alters_lab.services.snapshot_export import write_snapshot_yaml
from alters_lab.api.snapshot_intake import store
# retrieve confirmed session, export
path = write_snapshot_yaml(session.snapshot, Path('/tmp/demo_snapshot_export.yaml'))
print(f'Exported to: {path}')
"
```

**Pass**: YAML file exists at the explicit path. File contents match the canonical snapshot shape (`snapshot:` top-level key, all fields populated, `phase: "completed"`).

### Step 7: Show Active YAML Chain

Display the governed active YAML artifacts in `alters/current/`:

```

├── snapshot.yaml              ← confirmed snapshot
├── branches.yaml              ← branch discovery output
├── alters/
│   ├── alter_A.yaml
│   ├── alter_B.yaml
│   ├── alter_C.yaml
│   └── alter_D.yaml
├── value_alignment/
│   └── alignment_2026-05-19.yaml
├── dialogue/
│   └── dialogue_alter_D_2026-05-19.yaml
└── reality_trace.yaml         ← active reality trace
```

**Pass**: All files exist, are valid YAML, have `quality_status.human_confirmed: true` where applicable, and the chain is consistent (snapshot → branches → alters → value alignment → dialogue → reality trace).

**Pass**: 46 tests pass. Zero failures. Zero errors.

## 4. Pass Criteria

The Day 30 demo **passes** if and only if ALL of the following are true:

| # | Criterion | How Verified |
|---|-----------|--------------|
| P1 | API starts and `/health` returns 200 | `curl` or `httpx` call |
| P2 | Snapshot Intake session can be created via API | `POST /snapshot-intake/sessions` returns 201 |
| P3 | 3 anchors can be submitted in order via API | `POST .../answers` accepts all 3, rejects out-of-order |
| P4 | Snapshot confirms in-memory | `POST .../confirm` returns `phase: "completed"` |
| P5 | Export produces YAML at explicit target path | `write_snapshot_yaml(snapshot, path)` writes valid file |
| P6 | Exported YAML matches canonical snapshot shape | `snapshot:` top-level key, all fields present |
| P7 | Active YAML chain is complete and consistent | 9 artifacts in `alters/current/`, cross-references valid |
| P8 | Reality trace is active and references real evidence | `reality_trace.yaml` with `status: "active"`, evidence_refs point to real files |
| P9 | 46 pytest tests pass | `pytest -v` shows 46 passed |
| P10 | No calibration, drift, LLM, frontend, or database used | Audit trail: no `score_*.yaml` written, no provider config, no DB connection, no UI files |

## 5. Fail Criteria

The Day 30 demo **fails** if any of the following occur:

| # | Fail Condition | Why It Fails |
|---|----------------|--------------|
| F1 | Claiming a backend service exists that has no implementation code | Honesty violation — the system must not pretend capabilities it lacks |
| F2 | Active YAML artifacts are mutated without explicit human confirmation | Governance violation — all active YAML writes require human approval |
| F3 | Calibration scoring or drift computation is performed | Premature — reality trace `calibration_readiness.ready_for_reality_score: false` |
| F4 | An LLM provider is called or integrated | Out of scope for controlled first cycle |
| F5 | A database connection is established | In-memory only for Day 30 |
| F6 | A frontend or UI is demonstrated | No UI code exists; out of scope |
| F7 | The API confirms a snapshot and silently writes YAML | Export must be explicit — confirm stays in-memory |
| F8 | Tests fail or are skipped to make the demo pass | All 46 tests must pass genuinely |
| F9 | The demo references files or artifacts that don't exist on disk | Every evidence_ref must resolve to a real file |
| F10 | The reality trace claims events that didn't happen | Trace must only reference verifiable actions |
