4. Implement endpoints

Base path:
/snapshot-intake

Endpoints:
GET /snapshot-intake/health
Response: {"status": "ok", "component": "snapshot-intake"}

POST /snapshot-intake/sessions
Creates a new empty snapshot intake session.
Response: SnapshotSessionRead

GET /snapshot-intake/sessions/{session_id}
Returns current session.
404 if missing.

GET /snapshot-intake/sessions/{session_id}/next-anchor
Returns next required anchor and current phase.
404 if missing.

POST /snapshot-intake/sessions/{session_id}/answers
Body: {"anchor": "heaviest_constraint", "answer": "..."}
Behavior:
- Reject missing session with 404.
- Reject empty answer with 422 or 400.
- Reject answering an anchor that is not the current pending_anchor unless it was already completed and the service explicitly supports replacement. For P1-002, do not support replacement; reject out-of-order or duplicate answer.
- Update session snapshot through the existing pure service function.
- Return SnapshotSessionRead.

POST /snapshot-intake/sessions/{session_id}/confirm
Behavior:
- Only succeeds when all three anchors are answered and phase is ready_for_snapshot_confirmation.
- Calls mark_snapshot_completed.
- Returns SnapshotConfirmationResponse with ready_for_branch_discovery = true.
- If not ready, return 409 Conflict or 400 with clear error.
- Does not write YAML.
- Does not trigger Branch Discovery.

5. Preserve one-question-at-a-time rule

API must enforce this order:
1. heaviest_constraint
2. most_unclear
3. unwilling_to_give_up
4. confirm

Out-of-order answers should fail.
