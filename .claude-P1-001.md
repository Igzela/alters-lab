2. Define Snapshot schema in schemas/snapshot.py

Create enums:

IntakePhase:
- not_started
- asking_heaviest_constraint
- asking_most_unclear
- asking_unwilling_to_give_up
- ready_for_snapshot_confirmation
- completed

AnchorName:
- heaviest_constraint
- most_unclear
- unwilling_to_give_up

Create models:

SnapshotAnchors:
- heaviest_constraint: str = ""
- most_unclear: str = ""
- unwilling_to_give_up: str = ""

SnapshotContext:
- current_commitments: list[str] = []
- external_deadlines: list[str] = []
- energy_state: str = ""

SnapshotIntakeStatus:
- phase: IntakePhase = not_started
- completed_anchors: list[AnchorName] = []
- pending_anchor: AnchorName | None = heaviest_constraint

EvidencePolicy:
- source_mode: str = "self_report_only_phase0"
- confidence: str = "unset"
- unresolved_assumptions: list[str] = []

Snapshot:
- date: str | None = None
- version: int = 1
- anchors: SnapshotAnchors
- context: SnapshotContext
- intake_status: SnapshotIntakeStatus
- evidence_policy: EvidencePolicy
- notes: list[str] = []

Validation rules:
- version must be >= 1
- if intake_status.phase = completed:
  - all three anchors must be non-empty
  - completed_anchors must contain all three anchors
  - pending_anchor must be null
- if phase is not completed:
  - pending_anchor may be the next required anchor
- do not infer missing anchors
- do not auto-fill personal content

3. Define Snapshot Intake service in services/snapshot_intake.py

Implement pure functions only:

- create_empty_snapshot() -> Snapshot
- next_anchor(snapshot: Snapshot) -> AnchorName | None
- record_anchor_answer(snapshot: Snapshot, anchor: AnchorName, answer: str) -> Snapshot
- ready_for_confirmation(snapshot: Snapshot) -> bool
- mark_snapshot_completed(snapshot: Snapshot) -> Snapshot

Rules:
- These functions must not write files.
- These functions must not call external services.
- record_anchor_answer must reject empty answers.
- record_anchor_answer must update completed_anchors.
- record_anchor_answer must advance pending_anchor in this order:
  1. heaviest_constraint
  2. most_unclear
  3. unwilling_to_give_up
- After all three anchors are answered, phase should become ready_for_snapshot_confirmation.
- mark_snapshot_completed only succeeds if all three anchors are non-empty.
- mark_snapshot_completed sets phase = completed and pending_anchor = None.
- Do not generate branches.
- Do not generate alters.

4. Tests

Create pytest tests covering:

Schema tests:
- empty snapshot is valid
- completed snapshot with all anchors is valid
- completed snapshot with missing anchor is rejected
- version < 1 rejected

Service tests:
- create_empty_snapshot starts at not_started or asking_heaviest_constraint, but pending_anchor must be heaviest_constraint
- next_anchor returns heaviest_constraint first
- recording heaviest_constraint advances to most_unclear
- recording most_unclear advances to unwilling_to_give_up
- recording unwilling_to_give_up advances to ready_for_snapshot_confirmation
- ready_for_confirmation returns true after all three anchors are answered
- mark_snapshot_completed sets phase to completed
- empty answer rejected
- no branch or alter artifacts are created by service functions
